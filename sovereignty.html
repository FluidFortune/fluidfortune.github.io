THE GHOST ENGINE IS UNSEEN

The Ghost Engine runs on Core 0. Every interactive application — the AI terminal, the games, the notes editor, the file browser, the GPS app — runs on Core 1. This is not just a performance decision. It is an operational one.

Anyone watching the operator of a Pisces Moon device sees Core 1. They see a small handheld with a QWERTY keyboard. They see someone playing Snake, looking up a recipe in the field reference, writing a note, asking the AI a question. They do not see the Ghost Engine — because the Ghost Engine has no visible interface. It produces no notification. It writes to a thin status bar that only appears if the operator deliberately opens the WarDrive app. There is no badge, no toast, no LED. The device looks like a toy.

It is collecting the entire time. WiFi APs. BLE devices. GPS coordinates. Every wireless signature in range, written to the SD card with location and timestamp, while the operator does anything else.

This is the operational paradigm that distinguishes Pisces Moon OS from every other ESP32-S3 project. Bruce announces what it is doing. Meshtastic shows the operator the radio is active. Tactility loads the app the operator selected. Every other firmware on this hardware presents a coherent foreground state that an observer can read. Pisces Moon OS presents a coherent foreground state that lies — by design — about what the device is actually doing.

The Ghost Partition pairs with this directly. The captured data lives in a partition that is invisible in normal operation — no icon, no menu entry, no listing in the file browser. It requires a specific PIN combination at boot to even acknowledge it exists. With one of three PINs, the operator gets Tactical Mode (full access), Student Mode (a sanitized decoy partition that looks like a normal device), or the Nuke (sub-millisecond metadata deletion that renders the captured data permanently unreachable through any OS interface).

The device is always collecting. The operator can always destroy what was collected, instantly, on demand, without the destruction being visible. To a casual observer, an interrupted session, or a confiscation scenario, the device is a handheld toy that someone was playing chess on.

The Android deployment extends this argument further. A T-Deck Plus looks like a specialized field device — unusual enough that a trained observer might identify it. An Android phone looks like a phone. Everyone already ignores phones. The Pisces Moon Android APK runs the full wardrive intelligence suite as a background service while the operator plays Galaga, checks the weather, or watches a video in a regular Android app. A swipe exits the Pisces Moon interface entirely — the phone returns to a normal Android home screen. There is no visible indicator that a wardrive is running. There is no indicator that it was ever running. The most powerful cover for a field intelligence platform is the device everyone already trusts and already ignores.

Android's background task handling also removes the SPI Bus Treaty constraint entirely. On the T-Deck Plus, the Ghost Engine required inventing a named architectural protocol to run concurrent workloads on shared hardware without crashing. Android handles concurrent processes natively at the OS level. The wardrive runs as a background service. The foreground application does anything the operator needs. The OS manages the arbitration. The operator benefits from both the field capability and the most effective operational cover available: looking exactly like everyone else.

This is security through architecture, not through encryption. Encryption can be coerced. Architecture that hides the fact that there is anything to coerce is a different category of defense entirely.

PROBLEM 1 — AND THE INVENTION THAT SOLVED IT
The SPI Bus Conflict — and The SPI Bus Treaty
The Problem

The MicroSD card and the SX1262 LoRa radio module share the ESP32-S3's SPI bus. In an OS running the Ghost Engine and the LoRa mesh radio simultaneously, the conflict manifests as a hardware fatal exception — a Guru Meditation error and immediate device reboot with no user warning and no data recovery.

Why It Was Hard

The failure mode does not present as a bus conflict. The Guru Meditation dump points at a memory address identifying where the crash occurred — not why. The address changes between crashes because the timing of the collision is non-deterministic. This problem had no documented solution because no previous ESP32-S3 project had operated both devices under simultaneous sustained load.

THE INVENTION: THE SPI BUS TREATY

The SPI Bus Treaty is not a bug fix. It is a named architectural standard — the first known documented formal behavioral protocol governing shared-bus arbitration on the ESP32-S3 hardware class. It did not exist before this project. It now belongs to the public technical literature.

The Treaty consists of four rules that all OS components and all third-party ELF module developers must follow: Hit and run — open, write, close, release the bus immediately, no exceptions. No extended holds — no operation may hold the SPI bus for extended periods under any circumstances. Radio traffic management — a shared boolean flag coordinates WiFi radio access between the Ghost Engine and the AI client. Metadata-only destructive operations — the Nuke security function deletes index files in milliseconds rather than formatting the card over seconds.

These are not workarounds. They are laws. Every component in the OS is written to obey them. Every ELF module developer who builds for this platform is required to obey them. The Treaty is the reason Core 0 never stops — it is the architectural guarantee that makes persistent dual-core background tasking possible on shared-bus hardware.

The historical precedent is instructive: Unix filesystem locking conventions (1970s) established formal rules for competing processes sharing a filesystem. The Apollo Guidance Computer's priority scheduling protocol (1969) established formal rules for competing tasks sharing a processor under load. Nintendo's N64 RSP time budget (1996) established formal rules for competing subsystems sharing a signal processor. In each case, the solution was not a patch — it was a protocol. A named standard. A thing that could be taught, documented, and enforced across an entire platform.

The SPI Bus Treaty is that standard for the ESP32-S3. It is the first known documented instance of this solution class applied to this hardware architecture. Any developer who builds concurrent multi-subsystem software on the ESP32-S3 and encounters this problem now has a reference. They did not have one before.

The Treaty established a methodology. On ESP32-P4 hardware, the same methodology produced a formal UART peripheral map (four UARTs, four assigned roles, zero collisions) and a four-core convention (Ghost Engine on dedicated C6 silicon via SDIO, OS Core on HP Core 0, UI Core on HP Core 1, Sentinel Core on LP Core). Different hardware. Same governance principle. The methodology is the invention. The treaties are its expressions.

PROBLEM 2
Memory Exhaustion Under Simultaneous Workloads
The Problem

The ESP32-S3 has 320 kilobytes of fast internal SRAM. A general-purpose OS running a wardriving engine, a Gemini AI client, a BLE scanner tracking 100+ devices, a GPS reader, a LoRa mesh radio stack, and a 60fps user interface simultaneously exhausts internal SRAM. The device has 8MB PSRAM — but the heap allocator defaults to internal SRAM. PSRAM requires explicit configuration that had never been necessary before this project.

Why It Was Hard

SRAM exhaustion does not produce a single consistent error. Depending on which allocation fails, the system crashes in different subsystems — appearing as different bugs with different root causes until the pattern is recognized as a single systemic problem.

THE SOLUTION: PSRAM HEAP REDIRECTION

A single compiler flag — -DCONFIG_SPIRAM_USE_MALLOC=1 — automatically redirects heap allocations above a size threshold to PSRAM. The contribution is not discovering the flag — it is being the first to build something that required it and recognizing the failure pattern across apparently unrelated crashes.

PROBLEM 3
Dual-Core Task Synchronization
The Problem

Pisces Moon OS runs UI and applications on Core 1 and the Ghost Engine on Core 0. Concurrent SD card access from both cores corrupts SdFat's internal linked list data structures. The heap walker traverses the corrupted list, enters an infinite loop, and triggers the hardware watchdog timer.

Why It Was Hard

The failure is intermittent. In a bench environment with one WiFi network, collision is rare. In downtown Los Angeles with 80+ access points, the Ghost Engine writes to SD card multiple times per second and collision becomes near-certain during any concurrent Core 1 SD operation. The crash appears in random, unrelated locations. Root cause required correlating crash frequency with WiFi network density across multiple field sessions.

THE SOLUTION: MUTEX AND RADIO STATE FLAG

A FreeRTOS mutex created before either core starts. Every SD operation acquires the mutex with a timeout, executes, and releases immediately. A wifi_in_use boolean flag prevents the Ghost Engine and AI client from operating the radio simultaneously. Validated in downtown Los Angeles: 40+ simultaneous access points, continuous Ghost Engine logging, zero crashes after implementation.

PROBLEM 4
Dense RF Environment Instability
The Problem

In a real-world urban environment, 150+ Bluetooth devices may be simultaneously advertising. BLE callbacks arrive faster than the system can process them. Stack overflow. The overflow corrupts adjacent memory — appearing as GPS parsing errors, display glitches, random reboots — none of which indicate a BLE stack overflow.

Why It Was Hard

This problem cannot be discovered in a laboratory. The bench environment never exceeds twenty nearby devices. Downtown Los Angeles routinely exceeds one hundred. Three apparently unrelated symptoms — GPS timeouts, display corruption, random reboots — were all consequences of the same root cause: stack overflow propagating to adjacent data structures.

THE SOLUTION: STACK SIZING, SCAN BUFFERING, GPS DRAIN LOOPS

BLE task stack size increased for real-world urban RF density. Scan result buffer given a hard limit — new results discarded when full rather than crashing. GPS UART receive buffer increased from 128 to 512 bytes with explicit drain loops around the blocking WiFi scan call. All three fixes required simultaneously. Validated in downtown Los Angeles: sustained operation, no crashes, GPS fix maintained.

PROBLEM 5
The Security Architecture Constraint
The Problem

The Ghost Partition security system must render sensitive data unrecoverable quickly. The standard solution — AES encryption — holds the SPI bus for extended periods during write operations, triggering LoRa radio interrupt collisions via the same mechanism as Problem 1. The most widely deployed data security tool available causes the device to crash under this hardware architecture.

Why It Was Hard

This is not solvable within its original framing. If AES write operations violate the SPI Bus Treaty, and Treaty violations cause crashes, the requirement and hardware constraint are irreconcilable. The solution requires reframing the security requirement entirely: not "how do we encrypt the data" but "what does the adversary actually need to access the data."

THE SOLUTION: METADATA DELETION AND THREAT MODEL CALIBRATION

The OS maintains index files describing the structure and location of all Ghost Partition data. Deleting them takes milliseconds — a single small write completing within the Treaty's hold budget. Without the index files, all Ghost Partition data is permanently unreachable through any Pisces Moon OS interface.

Against the realistic threat model — casual inspection, card reader forensics without specialized equipment, time-constrained seizure scenarios — this provides complete protection. Against a Tier 3 adversary with laboratory-grade sector analysis capability, it provides meaningful delay. The threat model is documented explicitly and honestly. A device that crashes attempting to encrypt data is less secure than one that deletes index files in milliseconds and returns to normal operation.

PROBLEM 6
GPS Module Hardware Variation
The Problem

LilyGO manufactured the T-Deck Plus with two different GPS modules across production batches operating at different UART baud rates. Code that correctly initializes the GPS on one batch fails silently on the other. No error output. The GPS simply does not work. The variation is undocumented in official product documentation and not noted in any community forum.

Why It Was Hard

Silent failure without error output is the hardest category of embedded hardware problem to diagnose. The code is syntactically correct, compiles without warnings, executes without exceptions. Arriving at the hardware variation hypothesis requires eliminating every other possible cause first — a hypothesis that is difficult to generate without prior knowledge that this type of manufacturing variation occurs.

THE SOLUTION: BAUD RATE AUTO-DETECTION

An auto-detection system attempts GPS initialization at each supported baud rate in sequence, checks for valid NMEA sentence structure at each rate, and latches to the first configuration that produces valid output. Handles any future baud rate variants without requiring code changes. This is the first known documented solution to the T-Deck Plus GPS baud rate variation problem.

PROBLEM 7
GPIO Pin Collision Across Hardware Variants — The SD/Touch Conflict
The Problem

When Pisces Moon OS was ported to the T-LoRa Pager — a second LilyGO device sharing the same ESP32-S3 SoC — GPIO 21 was discovered to serve opposite functions on the two devices. On the T-Deck Plus, GPIO 21 is the touch controller reset pin (TOUCH_RST), wired to the GT911 capacitive touch IC. On the T-LoRa Pager, GPIO 21 is the SD card chip-select (SD_CS). Initializing the T-Deck touch driver on T-LoRa Pager hardware pulls SD_CS low at boot, corrupting the SD card mount before any application code runs.

Why It Was Hard

The same pin number, the same SoC, the same manufacturer — but completely different functions on two boards in the same product family. The failure mode was an SD mount corruption that appeared to be an SD driver problem, an SPI timing problem, or a power sequencing problem before the root cause was identified. No documentation explicitly maps pin functions across LilyGO product variants. Cross-referencing the schematic alone was insufficient; the labeling in the schematic PDF contained its own errors.

THE SOLUTION: CONDITIONAL COMPILATION WITH HARDWARE DEFINES

All touch initialization code is gated behind #ifdef DEVICE_TDECK_PLUS. On T-LoRa Pager targets, touch code is excluded at compile time — not disabled at runtime, which would still risk the GPIO state being set incorrectly during initialization. This is the first documented case of this specific GPIO collision between LilyGO T-Deck Plus and T-LoRa Pager hardware variants, and the first cross-device architecture comparison for Pisces Moon OS.

PROBLEM 8
SPI Bus State Corruption Across Peripheral Initialization — Deferred SD Mount
The Problem

On the T-LoRa Pager, SD card CMD0 — the initialization command — returned 0xFF (no response) eight times consecutively on every cold boot, despite the same physical SD card mounting successfully on T-Deck Plus hardware. The mount succeeded on warm reboot but not cold boot. The SD card was not defective. The SPI wiring was correct. The pin map was correct.

Why It Was Hard

Three independent causes converged to produce the same symptom. Each required separate diagnosis. Fixing one did not fix the problem. All three had to be resolved simultaneously: (1) An Arduino-ESP32 core 2.0.16 SPI/SD driver bug that was partially fixed in 2.0.17, requiring a platform version bump to espressif32 @ 6.10.0. (2) A wrong SPI pin map — the default Arduino SPI pins for the generic ESP32-S3 devkit are different from the physical SPI pins on the T-LoRa Pager PCB. The LilyGoLib variant header containing the correct pin map (SCK=35, MISO=33, MOSI=34, SS=21) had to be explicitly placed in the compiler include path. (3) A bus state problem — even with the correct core and correct pin map, the SD card refused to initialize because other SPI peripherals (display initialization, LoRa initialization) had left the SPI peripheral block in a state the SD card's CMD0 sequence did not accept.

THE SOLUTION: DEFERRED MOUNT WITH COLD SPI RESTART

SD initialization is deferred entirely out of the boot sequence into a background FreeRTOS task (late_sd_init_task) that runs after all other peripherals have completed their initialization. The task calls SPI.end(), waits 20ms for the bus to discharge, calls SPI.begin() with the correct pin map as a cold restart, then retries the SD mount up to ten times. The cold SPI restart eliminates the bus state problem. Apps poll a g_sd_ready flag with a timeout before accessing the SD card. The T-Deck Plus does not require this pattern because its peripheral initialization sequence happens to leave SPI in an SD-mountable state — a timing coincidence, not a design guarantee. The deferred mount pattern is now the recommended approach for any new ESP32-S3 hardware target.

All eight problems were present in the hardware. The first six were discovered because this project was the first software on the T-Deck Plus complex enough to trigger them. Problems seven and eight were discovered during the T-LoRa Pager port — hardware-variant collisions and bus state interactions that only surface when the same OS runs on two physically different devices. The solutions are part of the public technical record. The treaty doesn't change. The hardware around the treaty does.
