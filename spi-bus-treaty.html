THE SPI BUS TREATY

A structural problem with ESP32-S3 handhelds and the architectural contract that resolves it.

THE PATTERN

Three handhelds. Same SoC class. Same conflict.

LILYGO T-DECK PLUS
Display + microSD + SX1262 LoRa on one SPI bus
LILYGO T-LORA PAGER
Display + microSD + SX1262 LoRa + ST25R3916 NFC + nRF24 — all on one SPI bus
M5STACK CARDPUTER ADV + CAP LORA868
Display on its own SPI controller, but microSD and SX1262 LoRa share the second one
THE PATTERN
Every ESP32-S3 handheld with both LoRa and microSD has this conflict. This is not a product problem. It is a hardware class problem.

This is not a vendor mistake. The ESP32-S3 has three SPI peripheral controllers. The display already claims one for its own performance. The remaining two need to host everything else: storage, radios, sensors, expansion modules. Once you have more peripherals than controllers — and on any non-trivial handheld you will — peripherals share buses. Once peripherals share buses, they need arbitration.

The Pisces Moon SPI Bus Treaty is that arbitration, named and documented.

THE PHYSICS

What SPI is and what makes it brittle

SPI is a four-wire synchronous bus: clock, MISO, MOSI, and a per-peripheral chip-select line. Only one peripheral can be "selected" at any given moment. Selecting a peripheral pulls its chip-select low; deselecting it raises CS back to high-impedance. A transaction is a sequence of bytes shifted in and out while CS is held low.

The protocol has no concept of cooperation. There is no "I'm in the middle of something, please wait." There is no token. If task A pulls CS-A low to start a multi-byte read from the SD card, and task B pulls CS-B low to write a register on the LoRa radio while A's read is still in flight, the bus has now seen a clock pulse addressed to neither device in particular. Both devices respond unpredictably. Both transactions corrupt.

On hobbyist Arduino projects this rarely surfaces. There's usually one task, talking to peripherals sequentially, and the only "race" is between the main loop and an interrupt handler — small surface area, easy to spot.

On a handheld OS running multi-core with persistent background tasks, foreground apps doing arbitrary I/O, and radio interrupts firing on millisecond intervals, the surface area becomes large and the failure modes are subtle. Most of the time the bus survives because most transactions are short. Sometimes it doesn't.

SD cards are different

The two peripherals on the bus have very different timing characteristics, and this is where the structural problem really lives.

The SX1262 LoRa radio handles its bus transactions on the order of microseconds. A register read is 4 bytes. A FIFO write for a typical LoRa packet is 30–50 bytes. The radio also schedules its own internal work after a transaction completes, so the bus is free again almost immediately.

A microSD card handles its bus transactions on the order of milliseconds. A single 512-byte block read can take 1–2 ms under ideal conditions, longer under contention. Block writes can stall the bus for tens of milliseconds while the card's internal controller commits to flash. Mounting the card from a cold-power state can take hundreds of milliseconds — and on some cards, in some boot states, the card refuses to respond at all for the first several seconds, requiring repeated retries.

So when both devices share a bus, the timing mismatch is enormous. The radio assumes the bus will be free in microseconds. The card holds the bus for milliseconds. The radio's interrupt handler fires expecting to service an incoming packet, finds the bus busy with an SD operation it can't preempt, and either drops the packet or corrupts the SD transaction trying.

Without arbitration, the result depends on luck — which task got the bus first, how long it holds it, what the other task is trying to do. Some boots work. Some don't. Some logs are written. Some aren't. The failure mode is silent, intermittent, and impossible to reproduce.

EVIDENCE FROM THE FIELD

The Treaty's design wasn't a theoretical exercise. Each mechanism exists because a specific real-world failure surfaced during Pisces Moon's development across the three devices.

FAILURE 1
T-Deck Plus — Mesh Messenger drops LoRa packets during wardrive
Symptom

Sending a LoRa text message via Mesh Messenger while a wardrive session was active resulted in the message visibly being sent on the UI, but receiving nodes never got it. The wardrive log continued writing normally to SD.

Cause

Mesh Messenger's LoRa transmit path issued raw SPI transactions to the SX1262 without joining the Treaty mutex. The wardrive task held the bus during its SD writes; Mesh's transmit landed on a clocked-out bus addressed to a card that thought it was being read.

Fix

Mesh Messenger now takes spi_mutex before any LoRa operation. Recovered packet delivery, and as a bonus the wardrive session stopped getting its CSV writes mangled when LoRa packets were sent.

FAILURE 2
T-LoRa Pager — Wardrive silently writes nothing for entire sessions
Symptom

The Ghost Engine task started normally on boot, the wardrive UI showed WiFi networks being discovered and counts incrementing, the filename ribbon showed /wardrive_0015.csv — but pulling the SD card revealed no file by that name. The card had files from previous sessions (also empty) but nothing from the current boot.

Cause

Pager defers SD mount to a background task because the shared SPI bus can't always service the card during the first few seconds of boot. The wardrive task had a 15-second startup wait for g_sd_ready. If SD mounted after that window — which it did intermittently on this device — the task fell into a fallback path that constructed a timestamp-based filename and tried to append to it without ever calling sd.open(..., O_CREAT). SdFat silently dropped every subsequent write.

Fix — Lazy session-file creation (v2.7)

The task now leaves the filename empty until SD genuinely becomes ready, then claims the next available /wardrive_NNNN.csv slot. The Ghost Engine invariant — the device is always collecting — is restored on Pager regardless of mount timing. This bug doesn't exist on T-Deck Plus, where SD mounts synchronously during setup() before the wardrive task spawns. Same task, same code path, same Treaty discipline — different boot timing surfaces a different bug.

FAILURE 3
T-Deck Plus / T-LoRa Pager — File manager crash-loop
Symptom

Opening the on-device file manager would render the header strip and then crash-reset the device to the launcher. Sometimes it rendered a single row of files before crashing. Repeatable, but not always — about 60–70% of attempts.

Cause

The file manager's directory scan calls sd.open(directory) and dir.openNext() without taking the Treaty mutex. The wardrive task is writing CSV rows on the same bus at 4-second intervals. When the file manager's scan landed during a wardrive write, the resulting bus corruption either returned a null directory handle (which the code retried infinitely — the visible crash) or corrupted SdFat's internal state badly enough to trigger a watchdog reset.

Fix — Full Treaty discipline in filesystem.cpp (v1.2.0)

Wrap every SdFat call in the file manager with PM_SPI_TAKE / PM_SPI_GIVE. Use the sd_in_use sentinel flag to tell wardrive to pause its writes entirely while the file manager is active. Landed in v1.2.0. Ghost Ride The Whip — a read-only wardrive log inspector built with proper Treaty discipline — remains useful for verifying SD contents without the full file manager.

FAILURE 4
T-LoRa Pager — NRF24 power gating leaks current
Symptom

Pager battery life was ~30% worse than expected even with all radios apparently disabled. Profiling showed a 60 mA draw that didn't correspond to any active peripheral.

Cause

The Pager routes peripheral power through an XL9555 I²C GPIO expander. The NRF24 module shares a power rail with the LoRa radio. When apps disabled LoRa via PM_LORA_POWER_OFF(), the NRF24 was still being held in a partial-power state because its CS line was left floating from a previous transaction. The floating CS allowed phantom current draw through the chip's I/O protection diodes.

Fix

The Treaty's xl9555_boot_sequence() now explicitly drives NRF24 CS high before powering the rail, and drives it back to high-impedance only after the rail is fully off. This isn't a classic SPI race — it's a peripheral lifecycle issue that surfaces because the Treaty already owned the bus-state machine. Once you have a structured place to say "this is what the bus should look like in this state," lifecycle bugs become tractable.

THE TREATY'S DESIGN

The mechanism

A single FreeRTOS mutex (spi_mutex) plus a small set of conventions for using it.

extern SemaphoreHandle_t spi_mutex;

Defined once in wardrive.cpp, created during setup(), lives for the lifetime of the device. Every task that wants to issue SPI transactions to the shared bus must:

if (xSemaphoreTakeRecursive(spi_mutex, pdMS_TO_TICKS(500)) == pdTRUE) {
    // Bus is mine for the duration of this block
    FsFile file = sd.open("/log.csv", O_WRITE | O_APPEND);
    file.println(row);
    file.close();
    xSemaphoreGiveRecursive(spi_mutex);
}

The mechanism is plain FreeRTOS synchronization. The Treaty is the named convention around it. In v1.2.0 the mutex was converted to a recursive mutex, eliminating the entire class of nested-acquisition deadlock.

Named ownership

When debug logging is enabled, every take and give logs a human-readable owner name:

[TREATY] wardrive.scan      TAKE @ 12347ms
[TREATY] wardrive.scan      GIVE @ 12352ms
[TREATY] filemgr.opendir    TAKE @ 12353ms
[TREATY] filemgr.opendir    GIVE @ 12354ms

This is what lets you trace race conditions in field tests. When the SD writes vanish, the log shows you exactly who took the bus and never released it, or who tried to take it during someone else's window. Generic mutexes are anonymous; named ones tell you which subsystem broke discipline.

ISR-safe variants

Interrupt handlers — particularly the LoRa receive interrupt — need bus access but can't block. The Treaty provides:

#define PM_SPI_TAKE_ISR(pxWoken)
#define PM_SPI_GIVE_ISR(pxWoken)

These use xSemaphoreTakeFromISR() / xSemaphoreGiveFromISR() and pass through the higher-priority-task-woken pointer FreeRTOS needs. Without these, anyone porting a new interrupt handler would have to look up the right API surface; with them, it's copy-paste from existing examples.

Sentinel flags — cooperation, not just exclusion

The mutex protects the bus itself, but it doesn't tell other tasks why you have it. Sometimes you need a peer task to step back entirely while you're working, not just block waiting for the mutex. The Treaty defines sentinel flags:

sd_in_use — tells the wardrive task to skip its SD writes entirely while a file manager or diagnostic app is reading the card. Without this, the wardrive task would back up retries against the contended mutex and never make progress.

wifi_in_use — tells the wardrive task to skip its WiFi scans while another app needs the radio for its own purposes.

These are cheaper than additional mutexes (a single volatile bool read) and they express the right semantic: don't compete, defer.

Per-device extensions

T-LoRa Pager adds power-rail management via the XL9555 I²C GPIO expander. The Treaty extends to cover this:

PM_LORA_POWER_ON();    // raises XL9555 pin 0
PM_LORA_POWER_OFF();   // lowers XL9555 pin 0
PM_NFC_POWER_ON();     // raises XL9555 pin 1
// ...etc

These macros are no-ops on T-Deck Plus and Cardputer. On Pager, they wrap the I²C transactions that drive the rail and shadow the result in RAM so subsequent reads don't need a bus round-trip.

The Cardputer ADV with the Cap LoRa868 module is in the same category: SD and LoRa share FSPI, both need the Treaty. Cardputer's display SPI is separate (HSPI), which means display writes can happen at full speed without mutex contention. But anyone who writes a Cardputer app that touches SD or LoRa must still take the mutex. The Treaty is universal across the platform regardless of whether one specific bus is contended.

WHY THIS MATTERS BEYOND PISCES MOON

The structural problem is not specific to Pisces Moon, the LilyGO products, or the M5Stack ones. Any device built on ESP32-S3 with both LoRa and microSD will face this conflict. That's a hardware class, not a product.

Devices in the broader hardware ecosystem that fit this pattern:

LILYGO T-BEAM (ESP32-S3)
Same SoC, LoRa + SD on shared bus
HELTEC WIRELESS TRACKER
ESP32-S3, LoRa, storage — same class
HELTEC WIRELESS PAPER
ESP32-S3, LoRa, shared bus
M5STACK ATOM S3R + LORA ATOM
Modular attach, same arbitration problem
KODEDOT (announced Aug 2026)
Dual-chip architecture — explicitly designed to dodge this conflict by separating radio and storage onto different SoCs. The Treaty methodology still applies to shared resources between the two chips.
PISCES MOON ESP32-P4 PORT
Ghost Engine on dedicated C6 silicon via SDIO — same dual-chip architecture as KodeDot, built using Treaty methodology. The UART peripheral map is the P4 expression of the Treaty. The methodology scales with the hardware.
The Treaty pattern transfers directly — the mutex definition, the named ownership convention, the sentinel flags, the ISR-safe variants — all of it is portable to any ESP32-S3 device with shared-SPI peripherals. Naming it is the first step toward other projects in the same hardware class adopting something compatible.

WHAT THE TREATY IS NOT

It is not a custom mutex implementation. The mechanism is plain FreeRTOS. There's no novel synchronization primitive here.

It is not a substitute for proper SPI driver design. If the underlying ESP-IDF or Arduino-ESP32 SPI driver had bugs, the Treaty wouldn't fix them. The Treaty assumes the driver works correctly for any single transaction and arbitrates which task gets to issue them.

It is not a guarantee of bus health. A misbehaving peripheral that holds CS low past its transaction can still corrupt the bus regardless of mutex discipline. The Treaty makes the well-behaved case work correctly; it cannot rescue a malfunctioning peripheral.

It is not specific to the SX1262 or SdFat. Any pair of SPI peripherals with mismatched timing characteristics would exhibit the same pattern. The Treaty would apply unchanged to ESP32-S3 + NRF24 + SD, or ESP32-S3 + W5500 Ethernet + flash, or any other combination where one device wants the bus in microseconds and another holds it for milliseconds.

The novel contribution is the naming and the convention, not the code. Code is just a mutex. The contribution is recognizing that this is a cross-device structural pattern, giving it a name that travels with the code, building the debug logging and sentinel-flag conventions around it so the discipline is enforceable in code review, and documenting it so that anyone porting a new app or a new device knows the contract.

The Treaty is fully wired across Pisces Moon's three target devices as of v1.2.0. A systematic audit was performed for v1.2.0 — every call site walked, every exit path verified for proper release. Three real bugs were identified and fixed in the same release. The Treaty mutex was also converted to a recursive mutex in v1.2.0, preventing the entire class of nested-acquisition deadlock from ever surfacing again.

Apps that join the Treaty work. Apps that don't eventually break in field testing — sometimes subtly (wardrive writes silently dropping), sometimes obviously (file manager crash-loop). The pattern is enforceable because deviations are observable.

Without the Treaty, the Ghost Engine would be a marketing claim. With it, the Ghost Engine is a hardware-substantiated guarantee.
