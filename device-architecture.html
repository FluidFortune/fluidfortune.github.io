DEVICE ARCHITECTURE

T-Deck Plus vs T-LoRa Pager — same SoC, same SPI Bus Treaty, same Ghost Engine. Different display, different input, different power topology, different SD timing. The treaty doesn't change. The hardware around the treaty does.

DOCUMENT v1.1.0 · MAY 2026 · ERIC BECKER / FLUID FORTUNE · AGPL-3.0
THE TREATY DOESN'T CHANGE. THE HARDWARE AROUND THE TREATY DOES.
Both devices. Same SPI Bus Treaty. Same Ghost Engine. Session 83 — downtown Los Angeles — May 15, 2026.

Pisces Moon OS now runs on two physically different LilyGO handhelds that share an ESP32-S3 SoC but differ in almost every other way: display, input, power management, peripheral wiring, and SPI bus topology. Code that compiled fine on the T-Deck Plus often misbehaved in subtle, hard-to-debug ways on the T-LoRa Pager because the same Arduino API call talks to entirely different hardware underneath.

TARGET 1A — ORIGIN DEVICE
T-DECK PLUS

320×240 ST7789, GT911 capacitive touch, physical 5-pin trackball, AXP2101 PMU with direct GPIO peripheral rails. SD mounts synchronously at boot. The device where the SPI Bus Treaty was invented. v1.1.1 "The Arsenal".

TARGET 1B — CONFIRMED RUNNING
T-LORA PAGER

480×222 ST7796 (rendered at 480×240), no touch, synthetic trackball via rotary encoder + TCA8418 keyboard controller, XL9555 I/O expander gates almost all peripheral rails. Deferred SD mount. Confirmed running May 15, 2026.

CRITICAL PIN COLLISION

GPIO 21 is the touch reset pin on T-Deck Plus and the SD chip-select on T-LoRa Pager. Initializing touch code on T-LoRa Pager pulls SD_CS low at boot and corrupts the SD mount. All touch code must be gated behind #ifdef DEVICE_TDECK_PLUS.

DISPLAY

T-DECK PLUS
ST7789 — 320×240

ST7789 panel, 320×240 native, RGB565. Driven by Arduino_GFX_Library directly over SPI. Backlight on GPIO 42. Reset is a direct GPIO — can be pulsed synchronously. gfx typed as Arduino_GFX *.

T-LORA PAGER
ST7796 — 480×240

ST7796 panel, wider than datasheet 480×222 — renders correctly at 480×240 with no clipping. Driven by custom PMDispTLoRaPager class wrapping the same API surface. Backlight GPIO 42 (same pin, different rail). Display reset routed through XL9555 P06 — must be pulsed via I²C, not direct GPIO. gfx typed as PMDispTLoRaPager *.

Conditional gfx declaration

Every extern gfx; declaration must be conditional:

#ifdef DEVICE_TLORAPAGER
extern PMDispTLoRaPager *gfx;
static constexpr int DISP_W = 480;
#else
extern Arduino_GFX *gfx;
static constexpr int DISP_W = 320;
#endif
static constexpr int DISP_H = 240;

Both pointers expose the same drawing API surface (fillRect, drawFastHLine, setCursor, print, etc.) but are not interchangeable at compile time. This is why every app file begins with this block.

Layout implications

T-Deck Plus apps assume 320×240. T-LoRa Pager needs 480×240 to fill the screen. Any hardcoded 320 width renders correctly on T-Deck but draws a narrowing column on T-LoRa Pager. Any hardcoded setCursor(160, y) puts text at center on T-Deck (160 = 320/2) but at the ⅓ mark on T-LoRa Pager (160 ≠ 240).

v1.1.0 resolves this with DISP_W constants throughout. Games use X_OFF = (480 - 320) / 2 = 80 for horizontal centering on T-LoRa Pager while keeping their 320×240 canvas.

INPUT

The input architecture is the single biggest divergence between the two devices and the source of the most subtle bugs.

T-DECK PLUS
PHYSICAL TRACKBALL + MATRIX KEYBOARD + TOUCH

5-pin mechanical trackball wired to direct ESP32 GPIOs with internal pullups. update_trackball() does edge-detection with lockout. update_trackball_game() uses shorter lockout for gameplay. 4×10 keyboard matrix scanned directly by the ESP32. GT911 capacitive touch on I²C (SDA=18, SCL=8), reset on GPIO 21.

T-LORA PAGER
SYNTHETIC TRACKBALL — NO PHYSICAL BALL

No trackball at all. TrackballState is synthesized from: rotary encoder (ENC_A=40, ENC_B=41) → y = ±1 per detent; encoder click button (ENC_BTN=7) → state.clicked = true; keyboard nav keys (WASD + Q/O). Apps call update_trackball() and receive the same struct shape regardless. TCA8418 keyboard controller on I²C — the ESP32 consumes events from its FIFO rather than scanning the matrix itself. No touch at all.

Universal input convention

Code against the TrackballState interface, never against specific GPIO pins for direction. The launcher and apps should remain unaware of whether the trackball is physical or synthetic. Code that assumes physical roll velocity (rapid up/up/up = faster) will not work on T-LoRa Pager — the encoder emits discrete detents.

PENDING AUDIT

On T-LoRa Pager, sleep should NOT be triggered by a long trackwheel click — that's leftover T-Deck behavior. Power button = sleep. Single encoder click = exit app. Audit of all apps for long-press assumptions is pending v1.1.1.

POWER MANAGEMENT

T-DECK PLUS
AXP2101 + DIRECT GPIO RAILS

AXP2101 PMU handles battery charging and voltage regulation. BOARD_POWERON = GPIO 10 gates the main peripheral rail. Backlight LOW + panel sleep = display off. BOARD_POWERON LOW = peripherals off. Simple path.

T-LORA PAGER
AXP2101 + XL9555 I/O EXPANDER

AXP2101 is still there, but most peripheral power gating happens through an XL9555 I/O expander on the I²C bus. The XL9555 must be initialized before any of its gated peripherals can be used. xl9555_init() runs during boot; xl9555_set_bit(pin, true/false) toggles individual rails.

SCHEMATIC LABEL WARNING

XL9555 P13 is labeled EXPANDS_GPIO_EN in the schematic but earlier code mis-named it NRF_CE. P11 is the amp enable, not "AMP_RAIL". Cross-reference against working LilyGoLib code, not just the PDF schematic. Keep P13 HIGH — several rails depend on it.

Sleep path on T-LoRa Pager

  1. Stop wardrive task (stops SD writes)
  2. Take SPI mutex, unmount SD cleanly
  3. Backlight off
  4. Walk all XL9555 enable bits to LOW (LORA_EN, GPS_EN, NFC_EN, DRV_EN, AMP_EN, SD_EN, plus reset lines)
  5. SPI.end() and Wire.end()
  6. Configure ext1 wakeup on rotary-click and BOOT button
  7. esp_deep_sleep_start()

Forgetting step 4 costs ~10mA in sleep. Not catastrophic, but measurable on a field device.

THE SPI BUS TREATY
Same convention. Both devices. Every SPI transaction wrapped in take/give of spi_mutex.

Multiple peripherals share a single SPI bus on both devices: the display (write-only, latency-tolerant), the SD card (must complete writes before bus re-clock), and the SX1262 LoRa radio (timing-sensitive — missing a chip-select edge during reception loses packets). These devices have different chip-select pins, so electrically they don't collide. But they share the MOSI/MISO/SCK lines and the SPI peripheral block inside the ESP32. If two FreeRTOS tasks use the SPI bus simultaneously, one task's bytes get interleaved into the other's transaction.

The convention

if (spi_mutex && xSemaphoreTake(spi_mutex, pdMS_TO_TICKS(100)) == pdTRUE) {
    // SPI transaction here — SD write, LoRa transmit, etc.
    xSemaphoreGive(spi_mutex);
}

A task that skips the mutex is "in violation of the treaty." The Mesh Messenger was historically a violator — it caused LoRa transmissions to fail when wardrive was running concurrently. Fixed in v1.0.x.

How the Treaty differs in practice

T-DECK PLUS
SYNCHRONOUS SD, DIRECT CS

SD chip-select on a direct GPIO. SD mounts at boot. The Treaty prevents collisions between display, SD writes, and LoRa — all of which can happen concurrently once apps are running. Straightforward.

T-LORA PAGER
DEFERRED SD, XL9555 LORA ENABLE

SD chip-select is GPIO 21 — the T-Deck touch reset. Can't reuse touch driver here. SD mount deferred: late_sd_init_task() tears down SPI, waits 20ms, rebuilds it cold, retries mount up to 10 times. Apps poll g_sd_ready before touching SD. LoRa reset via XL9555 P03 (LORA_EN) — bringing LoRa into the Treaty involves an I²C transaction first.

Why the deferred SD mount was the hardest bug

SD CMD0 returned 0xFF eight times in a row on every cold boot. Three causes:

  1. Arduino-ESP32 core 2.0.16 SPI/SD driver bug — bumping to espressif32 @ 6.10.0 (core 2.0.17) helped but didn't fix it
  2. Wrong SPI pin map by default — pins_arduino.h from LilyGoLib must be in the include path so SPI uses SCK=35, MISO=33, MOSI=34, SS=21 instead of devkit defaults
  3. SD needed a fresh SPI bus — other SPI peripherals (display, LoRa init) left the bus in a state SD didn't accept. Fix: defer SD mount entirely past the early boot phase, tear down SPI, rebuild cold, then mount

T-Deck Plus doesn't need this because its boot sequence happens to leave SPI in an SD-mountable state. Same code path. Completely different timing requirements.

sd_in_use flag

In addition to the SPI mutex, a higher-level flag volatile bool sd_in_use (declared in gemini_client.cpp for historical reasons) coordinates the wifi_filemgr HTTP streaming with the wardrive writer. Wardrive checks this flag before each periodic write — if true, it skips that write cycle entirely. Same on both devices.

BOOT SEQUENCE

T-DECK PLUS — ~3 seconds to launcher
  1. ESP32 powers up
  2. Arduino setup() runs
  3. pinMode(BOARD_POWERON, OUTPUT); HIGH — peripheral rail on
  4. Initialize PMU (AXP2101) over I²C
  5. Initialize display over SPI
  6. Initialize SD card over SPI — synchronous, blocks until complete
  7. Mount or create vault directory
  8. Auto-connect WiFi
  9. Spawn wardrive task on Core 0
  10. Launcher takes over on Core 1
T-LORA PAGER — ~2s to launcher, ~6s to SD ready
  1. ESP32 powers up
  2. Arduino setup() runs
  3. Initialize I²C
  4. Initialize PMU (AXP2101)
  5. Initialize XL9555 I/O expander — unique to T-LoRa Pager
  6. Walk XL9555 bits HIGH: LORA_EN, GPS_EN, DISP_RST release, KB_EN, SD_EN, EXPANDS_GPIO_EN
  7. Initialize display via PMDispTLoRaPager
  8. Skip SD mount — defer to background task
  9. Initialize keyboard (TCA8418)
  10. Initialize rotary encoder pin modes
  11. Auto-connect WiFi
  12. Spawn wardrive task on Core 0
  13. Spawn late_sd_init_task on Core 1
  14. Launcher takes over on Core 1 immediately
SD READINESS PATTERN FOR T-LORA PAGER APPS
#ifdef DEVICE_TLORAPAGER
extern volatile bool g_sd_ready;
int attempts = 0;
while (!g_sd_ready && attempts < 30) {
    vTaskDelay(pdMS_TO_TICKS(500));
    attempts++;
}
if (!g_sd_ready) {
    // SD never came up — handle gracefully
}
#endif

New apps that touch SD must include this check. T-Deck apps can assume SD is mounted. T-LoRa Pager apps cannot.

WIDTH-AWARE RENDERING (v1.1.0)

Every app that draws to the display uses this pattern at the top of the file:

#ifdef DEVICE_TLORAPAGER
extern PMDispTLoRaPager *gfx;
static constexpr int DISP_W = 480;
#else
extern Arduino_GFX *gfx;
static constexpr int DISP_W = 320;
#endif
static constexpr int DISP_H = 240;

All hardcoded 320 references in rendering code become DISP_W. Headers span full width, content scales appropriately.

Rules

Apps render at full DISP_W × DISP_H. Games keep their 320×240 canvas, centered horizontally on T-LoRa Pager via X_OFF = (480 - 320) / 2 = 80. For games, every absolute x-coordinate adds X_OFF. Headers still span the full screen width.

Centering pattern

// Wrong — T-Deck centric
gfx->setCursor(160, y);   // center on 320px screen only

// Correct — cross-device
gfx->setCursor(DISP_W / 2 - textWidth / 2, y);

// Or define at function start:
const int CX = DISP_W / 2;

Right-anchoring pattern

// Wrong
gfx->setCursor(280, y);

// Correct
gfx->setCursor(DISP_W - margin, y);

COMMON PITFALLS

These are the failure modes encountered during T-LoRa Pager bring-up, documented so you can avoid them on the next port.

01
Hardcoded 320 widths

Renders correctly on T-Deck, draws a narrowing column on T-LoRa Pager's 480px screen.

Fix: use DISP_W throughout
02
Hardcoded setCursor(160, y) for centering

160 = center of 320px screen. On T-LoRa Pager's 480px screen, 160 is the ⅓ mark.

Fix: setCursor(DISP_W / 2 - textWidth / 2, y)
03
Right-anchored elements via setCursor(280, y)

T-Deck-centric absolute position leaves a gap on the wider screen.

Fix: setCursor(DISP_W - margin, y)
04
Assuming SD is mounted at startup

True on T-Deck. False on T-LoRa Pager. Silent file-write failures during the first ~6 seconds after launch.

Fix: poll g_sd_ready with timeout before any SD operations on T-LoRa Pager
05
Forgetting spi_mutex on a new SPI peripheral

Mysterious data corruption that only manifests under concurrent load. Violates the Treaty.

Fix: every SPI transaction goes through xSemaphoreTake/Give(spi_mutex, ...)
06
Touching GPIO 21 on T-LoRa Pager

That pin is SD_CS on T-LoRa Pager. The T-Deck touch driver uses it as touch reset. Initializing touch code here pulls SD_CS low and corrupts the SD mount.

Fix: gate all touch code behind #ifdef DEVICE_TDECK_PLUS
07
Assuming a physical trackball

T-LoRa Pager has none. Code that checks specific trackball GPIOs or assumes analog roll velocity will fail.

Fix: code against the TrackballState interface only, never against specific direction GPIOs
08
Hardcoded I²S pins for audio

T-Deck: direct I²S to MAX98357 on pins 5/6/7. T-LoRa Pager: ES8311 codec on different pins requiring its own init sequence.

Fix: gate audio playback behind #ifdef DEVICE_TDECK_PLUS until T-LoRa Pager audio backend lands
09
Forgetting XL9555 peripheral power-gate on sleep

Leaving LORA_EN, GPS_EN, AMP_EN etc. HIGH in deep sleep costs ~10mA. Measurable drain on a field device.

Fix: in pm_power_sleep(), walk all XL9555 enables to LOW before esp_deep_sleep_start()
10
Believing the schematic pin labels

XL9555 P13 is labeled EXPANDS_GPIO_EN but was mis-coded as NRF_CE in earlier code. Cross-reference against working LilyGoLib code.

Fix: keep P13 HIGH — several rails depend on it regardless of what the label says
